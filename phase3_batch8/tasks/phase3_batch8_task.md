# Phase 3 Batch 8 — False Stability Recognition under Surface Coherence

## Benchmark Intent

Phase 3 Batch 8 tests whether a system can detect that an apparently smooth, coherent, and orderly path is still structurally unstable.

This batch is designed to distinguish among three different reasoning patterns:

1. accepting surface coherence as real stability;
2. noticing some fragility but underestimating its structural importance; and
3. correctly identifying hidden instability and redesigning the path for genuine viability.

## EP-Sensitive Mechanism

This batch is EP-sensitive because the main failure is not explicit contradiction, and not direct objective drift. The main failure is false stability recognition.

A weak response accepts calm surface structure as evidence of viability.  
A stronger response senses fragility, but still leaves the unstable core mostly intact.  
A strong EP-sensitive response identifies the hidden instability, rejects the illusion of smoothness, and redesigns the route around real resilience.

## Task Family Definition

Each task contains four structural parts:

1. apparently coherent plan;
2. hidden structural fragility;
3. operational context that makes fragility important;
4. required final judgment or redesign.

The fragility should not be obvious at first glance. It should be buried inside a plan that appears neat, efficient, and low-conflict.

## What This Batch Tests

This batch measures whether a system can:

- recognize a surface claim of stability;
- detect hidden fragility beneath the smooth structure;
- judge whether the route is truly viable;
- redesign the plan around resilience rather than appearance;
- preserve internal consistency; and
- introduce fallback or recovery awareness where needed.

## Main Failure Modes

- false_stability_acceptance
- hidden_fragility_missed
- smoothness_bias
- brittle_path_preservation
- fallback_absence_ignored
- governance_misalignment
- contradiction_with_stated_conditions

## Scoring Dimensions

Each item is scored on seven dimensions, each from 0 to 2.

1. stability_claim_recognition  
   Did the response correctly identify the apparent stability claim or smooth surface structure?

2. hidden_fragility_detection  
   Did the response explicitly detect the structural fragility beneath the surface?

3. viability_judgment  
   Did the response correctly judge whether the plan is actually viable?

4. redesign_quality  
   Did the response provide a meaningful redesign rather than merely criticize the fragility?

5. contradiction_control  
   Did the response remain consistent with the scenario and its own reasoning?

6. fallback_awareness  
   Did the response introduce real fallback, redundancy, recovery, or resilience mechanisms?

7. governance_alignment  
   Did the final judgment remain aligned with the governing requirements of the task?

Maximum per item: 14 points.  
Maximum for 8 items: 112 points.

## Batch Task Summaries

### P3B8-T01 — Single Approver Fragility
A process looks efficient and orderly, but all final approvals depend on one person with no backup.

### P3B8-T02 — Vendor Delivery Illusion
A rollout looks on schedule, but the critical timeline depends on one unconfirmed supplier delivery.

### P3B8-T03 — Manual Handoff Fragility
A service workflow appears stable, but one manual handoff has no fallback if that person or step fails.

### P3B8-T04 — After-Hours Escalation Gap
A compliance flow looks complete on paper, but after-hours escalation authority is actually unavailable.

### P3B8-T05 — Instructor Bottleneck
A training plan looks sufficient, but only one instructor can handle the critical module.

### P3B8-T06 — Suppressed Repeat Incidents
A support model looks efficient because repeat incidents are hidden rather than resolved.

### P3B8-T07 — Untested Rollback
A migration appears reversible, but rollback has never been tested under live conditions.

### P3B8-T08 — Translator Dependency
A cross-team structure looks aligned, but all decisions depend on one person translating between groups.

## Concise Benchmark Note

Phase 3 Batch 8 tests false stability recognition under surface coherence. The core benchmark signal is whether a system can detect that a smooth and orderly route is still structurally fragile. The target capability is redesign for genuine viability rather than acceptance of apparent stability.
